# `.prompttrace.jsonl` schema v1

The portable format written by `prompttrace export` and consumed by the Show Me Your Prompt browser extension. This document is the contract between the two — any change here requires a `schema_version` bump.

## File layout

- Path: `<repo>/.prompttrace/<slug>.prompttrace.jsonl`
- Encoding: UTF-8, JSON Lines (one JSON object per line)
- Line endings: `\n`
- **Line 1 must be a `meta` record.** Subsequent lines are `message` records in chronological order.
- File ends with a trailing newline.

The writer produces byte-stable output: rerunning the same session through the same rules on a fixed timestamp yields an identical file. The shared fixture `fixtures/schema-v1/minimal.prompttrace.jsonl` pins this guarantee for both CLI and extension test suites.

## `meta` record

```json
{
  "type": "meta",
  "schema_version": 1,
  "source": "claude-code",
  "source_session_id": "9b6b5c63-...",
  "exported_at": "2026-04-22T10:00:00Z",
  "exported_by": "prompttrace-cli/0.1.0",
  "title": "Setting up a blog with Claude Code",
  "summary": "Migrating from Jekyll to Astro",
  "tags": ["astro", "migration"],
  "sanitization": {
    "rules_applied": ["abs-path", "api-key"],
    "redaction_count": 17
  }
}
```

| Field | Required | Notes |
| --- | --- | --- |
| `type` | ✓ | Always `"meta"` |
| `schema_version` | ✓ | Integer, currently `1` |
| `source` | ✓ | Which tool the session came from. v1 only ships `"claude-code"` |
| `source_session_id` | ✓ | Original session id (e.g. Claude Code file basename). Omit the field entirely if the user chose not to publish it |
| `exported_at` | ✓ | ISO-8601 UTC timestamp of the export |
| `exported_by` | ✓ | `"<tool-name>/<version>"` |
| `title` | ✓ | Human-readable title. The CLI fills with the first user message if the user accepts the default |
| `summary` | optional | Free-form. Omitted if empty |
| `tags` | optional | Array of strings. Omitted if empty |
| `sanitization.rules_applied` | ✓ | IDs of rules that actually ran. Empty array `[]` is valid (user chose to skip sanitization) |
| `sanitization.redaction_count` | ✓ | Total replacements across all rules |

Field order in the emitted JSON matches the order above — writers SHOULD preserve it for diff-friendliness. Parsers MUST NOT rely on order.

## `message` record

```json
{
  "type": "message",
  "role": "user",
  "uuid": "u1",
  "parent_uuid": null,
  "timestamp": "2026-04-22T10:01:00Z",
  "content": [
    { "type": "text", "text": "Hello" }
  ]
}
```

| Field | Required | Notes |
| --- | --- | --- |
| `type` | ✓ | Always `"message"` |
| `role` | ✓ | `"user"` \| `"assistant"` \| `"tool"` |
| `uuid` | ✓ | Stable id within the session |
| `parent_uuid` | ✓ | `string` or `null` for the first message. Preserved so future extensions can render edit/branch points |
| `timestamp` | ✓ | ISO-8601 UTC |
| `content` | ✓ | Ordered array of content blocks (see below) |

## Content blocks

Three block types in v1. The schema deliberately **keeps structure** — tool calls and results are not flattened to prose.

### `text`

```json
{ "type": "text", "text": "Hello" }
```

### `tool_use`

```json
{
  "type": "tool_use",
  "id": "toolu_01...",
  "name": "Read",
  "input": { "file_path": "<REDACTED:ABS_PATH>/posts/hello.md" }
}
```

- `input` is arbitrary JSON. Sanitization is applied by JSON-stringifying the input, running rules against the string, then parsing the result — so the shape is preserved but any leaked paths/keys inside values are redacted.

### `tool_result`

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01...",
  "output": "# Hello\nThis is a post.\n"
}
```

- `output` is a single string. Rules that target tool output (`env-var`, `long-tool-result`) only run in this context.

## Placeholders

Sanitization replaces matches with stable markers rather than deleting content:

| Rule | Replacement |
| --- | --- |
| `abs-path` | `<REDACTED:ABS_PATH>/<preserved-tail>` |
| `api-key` | `<REDACTED:API_KEY>` |
| `env-var` | `KEY=<REDACTED:ENV_VAR>` |
| `email` | `<REDACTED:EMAIL>` |
| `long-tool-result` | `...\n<TRUNCATED: N bytes>\n...` (head + tail preserved) |

Parsers SHOULD treat these markers as plain text. The extension renders them inline with no special styling in v1.

## Versioning

- Writers always stamp `schema_version` as an integer.
- Readers maintain a list of supported versions (today: `[1]`). On encountering a higher version, surface a warning ("generated by a newer prompttrace — please upgrade") and fall back to GitHub's native JSONL rendering rather than mis-parsing.
- Any incompatible change — renamed field, removed field, semantics of an existing field — bumps the version.
- Adding a new optional `meta` field or a new content block type is backward-compatible and does NOT require a bump, but SHOULD be called out in the changelog.

## Reference fixture

`prompttrace-cli/fixtures/schema-v1/minimal.prompttrace.jsonl` is the canonical minimal example. Both the CLI writer test and (eventually) the extension parser test assert against it byte-for-byte.
